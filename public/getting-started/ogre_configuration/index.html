      Getting Started
      Ogre configuration file

      Ogre configuration file

      Most ogre commands relies on ogre.yaml configuration file, It tells the CLI which parsers (plugins) to run, where to read source files,
      where to write outputs, and how the generated revords should be formatted.

      A ready to use ogre.yaml file is provided in the dfir-ogre/configuration/ folder

      Below you will find a complete description of every top‑level key, the supported sub‑structures, and the way the CLI reads the file.

      Overview

      lorem

      General Settings

      Key Description Example
      plugin_prefixes Prefix(es) that are automatically added to plugin class names when Ogre discovers parsers. - dfir_ogre_plugin
      output_folder Base directory where all generated artefacts are stored. $case is replaced at runtime with the case identifier. /data/operationnel/traitements/$case/
      report_folder Where the final HTML / PDF reports are written. It expands the output_folder with additional layout. /data/operationnel/traitements/$case/analysis/ogre_ng/ogre_report/$dir_tree
      temp_folder Temporary directory used while processing an ORC archive (e.g., to unpack files, hold intermediate YAML). /data/tmp/ogre_ng
      inner_archive_password Default password used for encrypted inner archives (if not overridden on the CLI). avproof
      plugin_folder Relative path (inside the configuration directory) that holds all XML plugin descriptors. /configuration

      These values are read directly by the CLI (see src/ogre/cli.py – handling of the --configuration argument) and later interpolated in the mapping section.

      Place‑holder Variables

      The configuration file uses a few runtime placeholders that are substituted by the CLI before the YAML is parsed:

      Placeholder What it expands to
      $output_folder The global output_folder defined at the top of the file (with $case already replaced).
      $plugin_folder The path to the folder that contains the XML plugin descriptors (/configuration).
      $mapping_label The mapping_label of the current mapping entry.
      $dir_tree Directory tree derived from the case (e.g., source host name).
      $computer_name The computer name extracted from the ORC metadata (report.computer).
      $file_name Base name of the source file (without extension).
      $case Replaced by the case identifier supplied on the CLI (e.g., --case mycase).
      $password Optional archive password passed from the CLI (not a placeholder in the YAML, but an argument).
      $TIMELINE_FOLDER$ Used only by the CLI to inject the timeline output directory when the user runs the timeline sub‑command.

      2. Output Definitions

      The output block defines reusable output “templates”.
      Each entry can be referenced from the mapping section via the output: list.

      2.1 Timeline output 

      dfir-ogre/configuration/ogre.yaml#L11-19
timeline:
    type: gzip
    format: jsonl
    date_format: iso_utc
    with_timeline: true
    with_qualifiers: false
    include_empty_field: false
    output_folder: $output_folder/analysis/ogre_ng/$mapping_label/$dir_tree/$computer_name
    base_file_name: $file_name
      • type – How the data is written on disk. "gzip" means the JSON‑Lines file will be compressed.
      • format – Serialisation format. jsonl = JSON‑Lines (one JSON object per line).
      • date_format – Timestamp representation (iso_utc = ISO‑8601 in UTC).
      • with_timeline – If true, the parser will include timeline information in each record.
      • with_qualifiers – When true, field names are suffixed with a qualifier (e.g., ip:src).
      • include_empty_field – If true, fields with no value are still emitted (useful for schema‑stable CSV).
      • output_folder – Destination directory for the generated file; it can use several placeholders (see § 4).
      • base_file_name – Name of the output file without extension; Ogre will add .jsonl.gz automatically.

      2.2 Fine (CSV) output 

      dfir-ogre/configuration/ogre.yaml#L21-29
fine:
    type: file
    format: normalized_csv
    date_format: iso_utc
    with_timeline: false
    with_qualifiers: false
    include_empty_field: true
    output_folder: $output_folder/fine/$mapping_label/$dir_tree/$computer_name
    base_file_name: $file_name
      • type: file – Plain (uncompressed) file.
      • format: normalized_csv – CSV with a stable column order; fields that are not present become empty strings.
      • Other flags behave exactly like the timeline block, but notice that with_timeline is false – the CSV does not contain timeline columns by default.

      Both blocks are later turned into an OutputConfiguration object (see src/ogre/configuration.py load_output_configuration).

      Mapping Section - Which plugin processes which artefact?

      The mapping list is the heart of the YAML. Each entry tells Ogre:

      • Which files (or archive entries) to match (original_file_pattern or archive_file_pattern).
      • Which XML plugin descriptor to load (plugin_file).
      • What label to give the output (mapping_label).
      • Which output definitions to use (output: list – references timeline and/or fine).

      Example entry (first three lines) 

      
- original_file_pattern: '.*\\ActivitiesCache.db$'
  plugin_file: $plugin_folder/activity_cache.xml
  mapping_label: activity_cache
  output:
      - timeline

      3.1 Pattern types (original_file_pattern / archive_file_pattern)

      Field Meaning
      original_file_pattern A regular expression applied to the original file path inside the ORC archive (e.g., C:\Users\…\ActivitiesCache.db).
      archive_file_pattern A regular expression applied to file names inside a sub‑archive (e.g., a CSV inside the ORC). Used for parsers that work on extracted CSV files.

      Only one of the two may be present per mapping entry.

      3.2 Plugin file lookup & $plugin_folder

      plugin_file points to a XML descriptor that defines how a plugin parses the matched artefact.
      The value is prefixed with $plugin_folder (set in the global section) – e.g.:

      plugin_file: $plugin_folder/amcache_file/amcache_program_wer.xml

      This will resolve at runtime to:

      /configuration/amcache_file/amcache_program_wer.xml

      3.3 Mapping label – output path stitching

      mapping_label becomes part of the destination path defined in the referenced output block:

      output_folder: $output_folder/.../$mapping_label/...

      Thus each parser’s results are stored in a dedicated sub‑directory named after its label, making it easy to locate artefacts later.
